# 大模型对话系统前后端开发方案

## 1. 项目目标

基于 `kaifa` 文档，建设一个可商用扩展的大模型对话系统，覆盖以下核心能力：

- 用户注册、登录、权限控制
- 会员体系与支付充值
- 多模型对话、历史记录管理、上下文管理
- 管理员后台：用户管理、套餐管理、模型管理、支付配置管理

本阶段仅输出开发方案，不涉及具体代码实现。

## 2. 需求拆解

### 2.1 用户端需求

- 用户名 + 密码注册/登录
- 注册时实时校验用户名是否重复
- 用户名重复时提供随机数追加建议，前端显示骰子按钮
- 注册时校验两次密码是否一致
- 普通用户可登录、查看套餐、充值、查看历史对话
- 只有会员用户才能与大模型进行对话
- 管理员拥有会员全部权益
- 聊天页支持：
  - 左侧历史对话列表
  - 右侧聊天内容区
  - 底部输入框
  - 输入框左侧模型选择
  - 上下文最多保留最近 5 轮

### 2.2 管理端需求

- 管理员登录后进入后台
- 用户管理：查看用户、角色、会员状态
- 套餐管理：金额、有效期、说明文字
- 模型管理：模型名称、调用 URL、API Key、启用状态
- 支付管理：配置支付接口参数，支持后续动态变更

### 2.3 后端核心规则

- 角色分级：
  - 普通用户
  - 会员用户
  - 管理员
- 权限规则：
  - 普通用户：不可发起大模型对话
  - 会员用户：可发起对话
  - 管理员：具备后台权限且可对话
- 聊天上下文：
  - 单会话最多保留最近 5 轮问答作为上下文
- 套餐配置变更后，用户端套餐展示需要动态更新

## 3. 总体技术方案

## 3.1 技术选型建议

### 前端

- 框架：`Next.js` + `React` + `TypeScript`
- UI：
  - 用户端：`Tailwind CSS` + 自定义聊天界面
  - 管理端：`Ant Design` 或 `Semi Design`
- 状态管理：`Zustand`
- 数据请求：`TanStack Query`
- 表单校验：`React Hook Form` + `Zod`

选择原因：

- `Next.js` 适合同时承载用户端和管理端，路由、鉴权、中后台页面管理都比较成熟
- `TypeScript` 有利于前后端接口约束统一
- 管理端使用成熟组件库可以显著降低开发成本
- 聊天页适合使用轻量状态管理与流式渲染

### 后端

- 框架：`NestJS` + `TypeScript`
- ORM：`Prisma`
- 数据库：`MySQL 8`
- 缓存：`Redis`
- 鉴权：`JWT` + `Refresh Token`
- 日志：`Winston` 或 `Pino`
- 接口协议：`RESTful API`
- 流式输出：`SSE`

选择原因：

- `NestJS` 适合中后台、权限、支付回调、模型路由、模块化开发
- `Prisma` 便于数据库建模与迁移
- `Redis` 可用于登录态、限流、上下文缓存、支付幂等控制
- `SSE` 实现大模型流式聊天比 WebSocket 更直接

### 部署

- 前端：`Vercel` 或 `Nginx + Node`
- 后端：`Docker` + `Node.js`
- 数据库：`MySQL`
- 缓存：`Redis`
- 文件与日志：本地挂载卷或对象存储

## 3.2 系统架构

推荐采用前后端分离 + 单后台服务架构。

```text
用户浏览器
  -> 前端 Web（用户端 + 管理端）
  -> API 网关 / 后端服务
      -> 用户认证模块
      -> 权限模块
      -> 会员模块
      -> 支付模块
      -> 聊天模块
      -> 模型路由模块
      -> 管理后台模块
      -> MySQL
      -> Redis
      -> 外部支付平台
      -> 外部大模型服务
```

首期建议使用单体模块化架构，不急于拆微服务。原因是需求聚焦、功能闭环强、交付效率更高。

## 4.3 关键交互设计

### 注册

- 用户输入用户名后，前端在失焦或停止输入 300~500ms 后调用用户名检查接口
- 如果用户名重复：
  - 输入框右侧展示骰子按钮
  - 点击后由前端请求随机建议用户名接口
  - 自动填入建议值并再次校验
- 两次密码输入完成后，前端即时提示是否一致
- 表单提交前再次做服务端校验

### 登录

- 使用用户名 + 密码
- 登录成功后根据角色跳转：
  - 管理员跳后台首页
  - 普通用户 / 会员跳聊天首页

### 聊天页

- 左侧显示会话列表，支持新建会话、切换会话
- 右侧显示消息流
- 底部输入区包含：
  - 模型选择
  - 文本输入
  - 发送按钮
- 非会员点击发送时：
  - 阻断实际调用
  - 弹出会员购买引导

### 套餐页

- 直接读取后台套餐配置
- 展示金额、有效期、说明
- 发起支付后显示订单状态

### 管理后台

- 采用左侧菜单 + 右侧内容区结构
- 各配置模块支持新增、编辑、启用/停用
- 支付配置单独集中管理，避免散落到环境变量之外不可维护

## 5. 后端开发方案

## 5.1 模块划分

建议后端拆为以下业务模块：

- `auth`：注册、登录、token 刷新
- `users`：用户信息、角色管理
- `memberships`：会员状态、到期时间、权益校验
- `packages`：套餐配置管理
- `payments`：支付下单、支付回调、订单状态更新、支付配置
- `models`：大模型配置管理、模型可用性控制
- `chat`：会话、消息、上下文裁剪、模型调用
- `admin`：后台管理接口聚合
- `audit`：操作日志、支付日志、模型调用日志

## 5.2 鉴权与权限方案

### 登录鉴权

- 登录成功返回：
  - `accessToken`
  - `refreshToken`
  - 当前用户信息
- 前端在请求头中带 `Bearer Token`
- 刷新令牌用于续签登录状态

### 权限模型

建议拆成两个维度：

- 角色 `role`
  - `user`
  - `admin`
- 会员权益 `membershipStatus`
  - `none`
  - `active`
  - `expired`

判断规则：

- 可访问管理端：`role = admin`
- 可发起聊天：`role = admin` 或 `membershipStatus = active`

这种拆法比直接定义“普通用户/会员/管理员”三选一更灵活，后续扩展更容易。

## 5.3 聊天模块设计

### 会话机制

- 用户可创建多个会话
- 每个会话绑定一个当前模型
- 每次发送消息时记录：
  - 用户问题
  - 模型回答
  - 消耗信息
  - 响应时间
  - 错误信息

### 上下文控制

- 最大保留最近 5 轮问答
- 1 轮 = 1 条用户消息 + 1 条模型回复
- 调用模型前：
  - 读取当前会话最近 10 条消息
  - 按时间顺序组装为上下文
- 如后续要支持更复杂策略，可扩展为：
  - system prompt
  - 摘要压缩
  - token 截断

### 模型调用抽象

统一封装模型适配层，不让业务层直接调用第三方模型：

- `ModelProvider` 接口
- 支持配置：
  - 模型名称
  - 请求 URL
  - API Key
  - 请求方法
  - 是否启用
- 聊天时按前端传入的模型 ID 查询配置，再由适配层发起调用

### 流式输出

- 推荐使用 `SSE`
- 后端边接收边转发模型输出片段
- 前端逐步拼接消息内容

## 5.4 支付模块设计

### 接入原则

根据你提供的支付平台文档进行集成，代码中将支付配置项做成后台可维护项，初期允许为空。

### 支付配置项建议

- 商户号
- 应用 ID
- 应用密钥
- 回调地址
- 支付网关地址
- 签名方式
- 是否启用沙箱

### 支付业务流程

1. 用户选择套餐
2. 后端创建本地订单，状态为 `pending`
3. 后端调用支付平台下单接口
4. 前端拉起支付
5. 支付平台异步回调后端
6. 后端验签成功后更新订单状态为 `paid`
7. 同时发放会员时长，更新会员到期时间

### 幂等与安全

- 支付回调必须验签
- 回调处理必须幂等
- 订单状态变更必须有流水日志
- 同一订单只允许成功发放一次会员权益

## 8. 安全与风控方案

## 8.1 认证安全

- 密码使用 `bcrypt` 或 `argon2`
- 登录接口增加限流
- JWT 设置合理有效期
- Refresh Token 存 Redis 或数据库

## 8.2 支付安全

- 支付回调必须验签
- 后端以回调结果为准，不信任前端支付成功状态
- 订单号唯一
- 回调接口防重放、防重复处理

## 8.3 模型调用安全

- 模型 Key 仅服务端存储
- 前端不直接暴露真实 URL 与 API Key
- 对聊天接口进行会员校验和频率限制

## 8.4 管理后台安全

- 管理员接口单独权限守卫
- 敏感操作写入审计日志
- 支付配置、模型 Key 等敏感字段加密保存

## 12. 风险点与提前规避

### 支付接入风险

- 第三方支付文档细节和验签规则可能复杂
- 建议先打通沙箱或测试环境，再接正式支付

### 模型兼容风险

- 不同模型接口格式可能不统一
- 必须先做统一适配层，避免业务代码与具体模型强绑定

### 权限边界风险

- 角色与会员状态容易混淆
- 建议严格拆为“角色”与“会员权益”两个字段

### 聊天成本风险

- 未做限流和调用统计可能导致成本不可控
- 建议记录 token 用量、响应时长、失败率

这个项目首期最合理的落地方式是：

- 前端使用 `Next.js + TypeScript`
- 后端使用 `NestJS + Prisma + MySQL + Redis`
