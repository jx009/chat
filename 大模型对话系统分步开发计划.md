# 大模型对话系统分步开发计划

## 1. 文档说明

本文档严格基于以下两份文档制定：

- 《大模型对话系统前后端开发方案》
- 《大模型对话系统数据库 ER 图与接口文档》

目标不是再讲一次架构，而是明确实际开发顺序。后续开发必须按本计划逐步推进，每一步完成并验收后，再进入下一步。

## 2. 开发总原则

- 严格使用既定技术栈：
  - 前端：`Next.js + React + TypeScript + Tailwind CSS + Zustand + TanStack Query`
  - 后端：`NestJS + TypeScript + Prisma + MySQL + Redis + JWT + SSE`
- 严格按既定数据模型开发，不随意增删核心表
- 严格按接口文档命名接口、字段、状态值
- 严格区分两个概念：
  - `role`：`user/admin`
  - `membershipStatus`：`active/expired/none`
- 聊天权限判断必须按文档执行：
  - `admin` 可聊天
  - 有效会员可聊天
  - 普通非会员不可聊天

## 3. 开发顺序总览

建议按以下 12 步推进：

1. 项目初始化与工程骨架
2. 数据库建模与基础环境接通
3. 认证与用户体系后端开发
4. 登录注册前端开发
5. 管理后台基础框架开发
6. 套餐管理与支付配置开发
7. 支付下单、回调与会员权益发放
8. 模型管理开发
9. 会话与消息系统开发
10. 聊天调用链路开发
11. 权限联动、联调与异常处理
12. 测试、部署与上线准备

下面是每一步的详细内容。

## 第三步：认证与用户体系后端开发

### 目标

### 具体任务

开发以下接口：

- `GET /api/auth/suggest-username`
- `POST /api/auth/register`
- `POST /api/auth/login`

- `GET /api/users/me`

#### 认证能力

- 用户名唯一校验
- 用户名随机推荐
- 两次密码一致性校验
- 密码哈希存储
- JWT 生成
- Refresh Token 管理
- 登录信息写入 `last_login_at` / `last_login_ip`

#### 权限基础

- JWT 鉴权守卫
- 管理员角色守卫
- 当前用户解析装饰器

### 产出物

- 认证模块
- 用户模块
- 鉴权中间件
- 用户信息返回结构

### 验收标准

- 用户能注册成功
- 用户名重复时能正确报错

- 登录后能返回 token

#### 登录态管理

### 产出物

## 第七步：支付下单、回调与会员权益发放

### 目标

打通充值闭环，让用户完成支付后真正获得会员身份。

### 具体任务

#### 后端接口

开发：

- `POST /api/payments/orders`
- `GET /api/payments/orders/:id`
- `POST /api/payments/callback`
- `GET /api/memberships/me`

#### 支付逻辑

- 创建本地订单
- 调第三方支付下单
- 保存第三方订单号
- 支付回调验签
- 幂等更新订单状态
- 发放会员权益
- 延长会员有效期

#### 前端

- 套餐下单按钮
- 支付状态页 / 状态提示
- 支付成功后刷新会员状态

### 会员发放规则

- 若用户当前无有效会员：
  - `start_at = 当前时间`
  - `expire_at = 当前时间 + 套餐时长`
- 若用户当前仍为有效会员：
  - 从原 `expire_at` 基础上顺延

### 产出物

- 订单模块
- 支付回调模块
- 会员状态更新逻辑
- 充值流程页面

### 验收标准

- 用户能创建订单
- 回调成功后订单状态变为 `paid`
- 会员状态正确更新
- 会员到期时间按规则顺延

## 第十步：聊天调用链路开发

### 目标

在已有用户、会员、模型、会话、消息基础上，真正打通对话能力。

### 具体任务

#### 后端接口

开发：

- `POST /api/chat/send`
- `GET /api/chat/stream/:messageId`

#### 核心逻辑

- 校验用户聊天权限
- 校验会话归属
- 校验模型可用性
- 查询最近 5 轮上下文
- 写入用户消息
- 创建 assistant 消息占位
- 调用模型接口
- 更新消息内容、token 消耗、耗时、状态
- 支持 SSE 流式输出

#### 模型适配层

- 统一封装模型调用器
- 不允许业务层直接写死第三方请求
- 后续支持多模型扩展

#### 前端

- 输入框
- 模型选择器
- 发送按钮
- 流式消息渲染
- 消息生成中状态

### 上下文规则

- 单会话最大使用最近 5 轮对话作为上下文
- 当前用户输入必须追加到上下文末尾
- 后续如需要可增加 `system prompt`

### 产出物

- 聊天发送链路
- SSE 流式返回
- 上下文拼接逻辑
- 前端聊天交互

### 验收标准

- 会员用户可以正常聊天
- 非会员用户被正确拦截
- 管理员可以正常聊天
- 会话消息能正确保存
- 上下文严格只保留最近 5 轮

- 支付失败提示
- token 失效自动跳转登录
- 会话不存在或模型不可用提示

### 产出物

- 权限联动完整闭环
- 错误码处理
- 异常提示规范

### 验收标准

- 各模块流程串联无断点
- 非法访问都能被拦截
- 主要异常场景都有用户提示

## 5. 当前建议

后续实际编码时，不要同时多线展开，应该严格按顺序推进。

最合理的启动方式是：

1. 先执行第一步：项目初始化与工程骨架
2. 紧接着执行第二步：数据库建模与基础环境接通
3. 然后进入第三步和第四步，先把登录注册闭环做出来

这样推进，风险最低，返工最少。
